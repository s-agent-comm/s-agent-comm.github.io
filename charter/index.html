Goals

The Semantic Agent Communication Community Group aims to define semantic models and communication structures that enable autonomous software agents to interoperate in a verifiable, accountable, and machine-interpretable manner. Existing approaches focus on transport protocols or model-level behaviors. This group focuses instead on the semantic layer required for identity, delegation, capability discovery, intent modeling, and execution traceability. The group’s goal is to establish a common semantic foundation for interoperable agent-to-agent communication across heterogeneous systems, identity frameworks, and governance requirements.

Scope of Work

The Community Group will define and maintain the formal ontology for AI Agent interoperability, security, and communication. This includes a machine-readable, logically consistent framework for defining agents, their capabilities, and the rules governing their interactions. The scope includes:

  • Interaction Semantics: Formal definitions for Intent, Delegation, Capability advertisement, ExecutionRecord, and related CommunicativeActs.
  • Identity and Verifiability Primitives: Use of Decentralized Identifiers (DIDs), Verifiable Credentials (VCs), signatures, and provenance structures needed to establish trust in agent actions. This includes binding DIDs to hardware roots of trust (e.g., TEEs like ARM TrustZone) through attestation.
  • Narrative Model: The definition of an append-only, context-complete semantic log that provides traceability, accountability, and explainability of agent behavior.
  • Semantic Ledger Model: A logical model for representing immutable, ordered, verifiable records. This is a semantic abstraction—no blockchain or specific implementation technology is implied.
  • Conformance Definitions: Minimal normative requirements for interpreting CommunicativeActs, Delegations, and other semantic elements in a consistent manner across implementations.
  • Machine-Readable Ontology Artifacts: RDF/OWL vocabularies, JSON-LD contexts, SHACL shapes that express the above concepts in a formally verifiable way.
  • AgentIDL (WebIDL extension): An Interface Description Language generated from the core ontology to facilitate code generation. The scope includes defining the language and producing a reference implementation of an AgentIDL compiler.
  • Knowledge Representation Technology Bridge for Agent Memory: Defining the AgentIDL interfaces and serialization processes for integrating stateful, version-controlled knowledge representation technologies (e.g., GraphDBs, systems supporting version control like Git for knowledge artifacts) as a bridge for an Agent's internal memory and communication. Note: Stateless memory solutions like Vector Databases are explicitly out of scope, as they do not support the cognitive memory requirements for agents.
  • Schema.org Extension Submission: Developing and submitting extensions to Schema.org to incorporate Agent-specific semantic concepts, facilitating broader web discoverability and interoperability.

Out of Scope

The following topics are explicitly not within the scope of this Community Group. These topics may be implemented by downstream systems, but will not be defined, standardized, or constrained by this CG:

  • Transport Protocols: No definition or modification of any transport layer (e.g., HTTP, WebSockets, TCP, UDP, QUIC).
  • Internal Agent Reasoning Models: The CG does not define LLM behavior, planning algorithms, inference engines, or internal cognitive architectures.
  • Execution Backends and Compute Targets: The CG does not specify how semantic instructions are executed on CPU, GPU, WASM, or other hardware.
  • Economic Models and Payment Systems: The CG does not define settlement systems, payment rails, or token models.
  • Blockchain / Distributed Ledger Technology: The CG does not mandate or define consensus protocols or DLT implementations.
  • Domain-Specific Knowledge or Vertical Ontologies: No legal, medical, financial, or other industry-specific knowledge models are defined.
  • Governance, Policy, or Legal Interpretation: The CG does not define regulatory frameworks or legal norms.
  • Runtime Implementations or SDKs: The CG does not produce executable software, agents, or runtimes.
  • Conflict Resolution Mechanisms: The CG does not define mechanisms for resolving conflicts (e.g., CRDT, OT).
  • Browser Applications: The CG does not focus on browser-based applications or short-lived client-side interactions.

Deliverables

Specifications

The Community Group intends to publish:

  • AI Agent Semantic Communication Ontology v1.0: Published in Turtle and JSON-LD, encompassing all defined modules.
  • AgentIDL Specification and Reference Implementation: A formal specification for AgentIDL and a reference implementation of its compiler.
  • SHACL Shapes for Ontology Validation: A comprehensive suite of SHACL shapes to ensure data quality and adherence to the ontology.
  • Implementation and Governance Notes: Guidelines and best practices for implementing and governing agents based on the ontology.
  • Interoperability Best Practices: Guidelines for how agent runtimes, OS-level systems, and model providers can adopt the ontology for seamless interoperability.

Non-Normative Reports

The group may produce additional Community Group Reports such as use cases, requirements, white papers, interoperability guides, and mapping documents.

Test Suites and Other Software

The group MAY produce test suites and example implementations to support the specifications. See the GitHub LICENSE file for contribution licensing information.

Dependencies or Liaisons

The group expects to coordinate with the following:

  • Credentials Community Group
  • Decentralized Identifier Working Group
  • Verifiable Credentials Working Group
  • Web of Things Working Group
  • AI governance and safety-related community initiatives
  • External agent framework communities and implementers

Community and Business Group Process

The group operates under the Community and Business Group Process. Terms in this Charter that conflict with that Process are void.

W3C seeks organizational commitments under the W3C Community Contributor License Agreement (CLA). When an individual contributes without representing an organization, W3C will seek a CLA from the relevant organization at the point the individual contributes to a Deliverable.

The W3C Code of Ethics and Professional Conduct applies to participation in this group.

Work Limited to Charter Scope

The group will not publish Specifications outside the scope stated above. See Amendments to this Charter for the process to revise the scope.

Contribution Mechanics

Substantive contributions to Specifications may only be made by participants who have agreed to the W3C Community Contributor License Agreement (CLA).

Specifications and software outputs must use the W3C Software and Document License.

All contributions are made via the group’s GitHub repositories. Contributions may be submitted via pull requests, issues, or comments on existing issues.

All repositories must contain the standard CONTRIBUTING and LICENSE files.

Transparency

All technical work will occur in public GitHub repositories. Meeting minutes and summaries will be posted publicly.

Decision Process

The group will seek consensus. If consensus is unclear, a Call for Consensus (CfC) may be issued. When required for progress, Committers may select an alternative with substantial support. Objections may be addressed by maintaining alternative forks for evaluation.

Chair Selection

Participants choose their Chair(s). If five participants from distinct organizations call for an election, the group will run a formal election as described in the charter template, consulting the Community Development Lead as necessary.

Amendments to this Charter

The group may propose an amended charter using the Decision Process. Approval requires a 30-day vote and 2/3 support of votes cast. Minor corrections may be made through simpler group decisions.
